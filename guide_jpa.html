JPA — Java Persistence API

Come mappare un modello concettuale in entità JPA, scegliere le annotazioni giuste, decidere fetch type e cascade. Tutto con esempi reali dai TDE.

Cos'è JPA?

JPA (Java Persistence API) è uno standard Java che permette di mappare automaticamente le classi Java sulle tabelle del database, senza scrivere query SQL a mano. Le classi Java diventano entità, i loro attributi diventano colonne, e le relazioni tra classi diventano foreign key (o tabelle di join).

Il persistence context è un'area di memoria gestita dall'EntityManager: tiene traccia di tutte le entità "gestite" (managed) nella sessione corrente. Se carichi la stessa riga due volte, ottieni la stessa istanza Java (identità garantita). Le modifiche alle entità managed vengono rilevate automaticamente (dirty checking) e scritte al commit.

1. @Entity e @Id — Le basi

@Entity  // dice a JPA che questa classe è una tabella nel DB
public class Author implements Serializable {

  @Id  // chiave primaria
  @GeneratedValue(strategy = GenerationType.IDENTITY)  // autoincrement
  private int id;

  private String firstName;  // colonna normale
  private String lastName;

  @Temporal(TemporalType.DATE)  // per campi data
  private Date birthDate;

  @Basic(fetch = FetchType.LAZY)
  @Lob  // Large Object: per file grandi (foto, documenti)
  private byte[] photo;
}
Regola

Ogni entità JPA deve avere:
1. @Entity sulla classe
2. @Id su un campo (la chiave primaria)
3. Implementare Serializable (buona pratica)
4. Un costruttore senza argomenti (anche se non scritto, Java lo genera)

2. Tipi di relazione

Le relazioni tra entità rispecchiano le cardinalità del modello ER:

AnnotazioneCardinalitàEsempio
@OneToOne1:1Persona ↔ Passaporto
@ManyToOneN:1Libro → Autore (molti libri, un autore)
@OneToMany1:NAutore → Lista<Libro>
@ManyToManyN:MStudente ↔ Corso
Come scegliere l'annotazione giusta

Leggila dal lato dell'entità corrente:
"Un Autore può avere molti Libri" → nella classe Author: @OneToMany su List<Book>
"Un Libro ha uno Autore" → nella classe Book: @ManyToOne su Author

3. Owner side e mappedBy — La parte più difficile

In JPA, per ogni relazione bidirezionale, uno dei due lati è il proprietario (owner) della relazione. L'owner è il lato che "controlla" la foreign key nel database (cioè il lato che ha la colonna FK o la tabella di join).

Il lato non-owner deve specificare mappedBy = "nomeAttributo" dove nomeAttributo è il nome dell'attributo nell'entità owner che punta a questa relazione.

Esempio: Libro - Autore (Many-to-One)

La tabella Book ha la colonna FK author_id. Quindi Book è l'owner della relazione.

// Book è l'OWNER: ha la foreign key
@Entity
public class Book {
  @ManyToOne
  @JoinColumn(name = "author_id")  // nome della FK nel DB
  private Author author;  // owner
}

// Author è il lato NON-owner: usa mappedBy
@Entity
public class Author {
  @OneToMany(mappedBy = "author")  // "author" = nome campo in Book
  private List<Book> books;
}
Regola pratica per scegliere l'owner
  • @ManyToOne: sempre l'owner è il lato "Many" (ha la FK)
  • @ManyToMany: puoi scegliere qualsiasi lato come owner, preferisci il lato più acceduto o quello che rappresenta l'associazione
  • @OneToOne: l'owner è il lato che ha la FK (spesso quello con la dipendenza funzionale)

4. @JoinColumn e @JoinTable

@JoinColumn specifica il nome della colonna FK nel database. Va sul lato owner.

@ManyToOne
@JoinColumn(name = "director_id")  // nome colonna FK nel DB
private Artist director;

@JoinTable si usa per le relazioni @ManyToMany: specifica la tabella di join intermedia e le sue colonne.

@ManyToMany
@JoinTable(
  name = "actor_movie",              // nome tabella di join
  joinColumns = {@JoinColumn(name = "movie_id")},         // FK verso questa entità
  inverseJoinColumns = {@JoinColumn(name = "artist_id")}  // FK verso l'altra
)
private List<Artist> actors;

5. LAZY vs EAGER — Come vengono caricati i dati

Fetch TypeQuando vengono caricati i dati collegatiProContro
EAGERSubito, insieme all'entità principaleDati sempre disponibiliCarica dati forse non necessari (inefficiente con tanti dati)
LAZYSolo al primo accesso (on demand)Efficiente: carica solo quello che serveRichiede sessione aperta al momento dell'accesso
LazyInitializationException — L'errore più comune in JPA

Se hai una relazione LAZY e provi ad accedervi dopo che la sessione è chiusa, ottieni LazyInitializationException. Soluzione: mantieni la sessione aperta, oppure usa EAGER per quelle relazioni critiche.

6. Default fetch types — Cosa usa JPA se non specifichi

AnnotazioneDefault fetchMotivazione
@OneToOneEAGERDi solito è un singolo oggetto, pochi dati
@ManyToOneEAGERCome sopra: un singolo oggetto padre
@OneToManyLAZYPotenzialmente molti elementi: meglio caricare su richiesta
@ManyToManyLAZYCome sopra
Regola per l'esame: decidi in base alla cardinalità e all'uso
  • Se la collezione ha pochi elementi (es. premi di un artista → ordini di grandezza: decine) → EAGER va bene
  • Se la collezione ha molti elementi (es. tutti i reader di un libro → milioni) → LAZY obbligatorio
  • Se l'applicazione accede sempre quella relazione nella stessa request → EAGER è più efficiente

7. Cascade — Propagare le operazioni

Cascade dice a JPA di propagare automaticamente le operazioni JPA (persist, remove, merge, refresh) dall'entità padre ai figli.

CascadeTypeCosa propagaUso tipico
PERSISTEntityManager.persist()Salva automaticamente le entità figlio quando salvi il padre
REMOVEEntityManager.remove()Cancella i figli quando cancelli il padre (attenzione: pericoloso)
MERGEEntityManager.merge()Aggiorna i figli quando aggiorni il padre
REFRESHEntityManager.refresh()Ricarica i figli dal DB quando ricarichi il padre
ALLTutte le operazioniUsa con cautela: propaga tutto
Quando NON usare CASCADE.REMOVE

Se l'entità figlia può esistere indipendentemente dal padre (es. un Libro esiste anche senza Author), non usare REMOVE. Lo usi solo se il figlio non ha senso senza il padre (es. righe di un ordine d'acquisto).

8. Metodo passo-passo per l'esame JPA

Leggi il diagramma ER e individua le entità e le relazioni

Ogni rettangolo del diagramma = una classe Java con @Entity. Ogni rombo = una relazione (OneToMany, ManyToMany, etc.).

Determina la cardinalità di ogni relazione

1:1, 1:N, N:M? Questo ti dice quale annotazione usare.

Identifica quali direzioni della relazione servono all'applicazione

L'applicazione mostra "per un autore, i suoi libri"? → Servono Book nella classe Author. "Per un libro, il suo autore"? → Serve Author nella classe Book. Se serve solo una direzione, non è necessario il lato inverso (unidirezionale).

Scegli l'owner e metti mappedBy sul lato non-owner

Per ManyToOne: sempre Many è owner. Per ManyToMany: scegli il lato "più naturale" (spesso quello che ha la joinTable nel DB).

Scegli il fetch type in base a cardinalità e frequenza di accesso

Molti elementi → LAZY. Pochi sempre necessari → EAGER. Guarda le cardinalità dell'applicazione (migliaia? milioni?).

Decidi se serve @OrderBy per le liste ordinate

Se l'applicazione mostra elementi "in ordine decrescente di data" → aggiungi @OrderBy("date DESC").

9. Esempio completo — Digital Library (TDE Settembre 2025)

Schema: Book, Author, Reader, Country. L'applicazione mostra autori con i loro libri, e per ogni libro tutti i reader. Reader: milioni, Book: decine di migliaia, Author: migliaia, Country: decine.

@Entity
public class Author implements Serializable {
  @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
  private int id;
  private String name;
  private String surname;

  // Country → Author: ManyToOne (molti autori per paese)
  @ManyToOne(fetch = FetchType.LAZY)  // LAZY: non sempre serve il paese
  @JoinColumn(name = "country_id")
  private Country country;  // owner

  // Author → Books: OneToMany. App: "per un autore, i suoi libri"
  @OneToMany(mappedBy = "author", fetch = FetchType.LAZY)
  // LAZY: decine di migliaia di libri totali, su richiesta
  private List<Book> books;
}
@Entity
public class Book implements Serializable {
  @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
  private int id;
  private String title;

  // Book → Author: ManyToOne (owner: Book ha la FK)
  @ManyToOne(fetch = FetchType.EAGER)  // EAGER: l'autore è sempre mostrato con il libro
  @JoinColumn(name = "author_id")
  private Author author;

  // Book → Readers: ManyToMany. App: "per ogni libro, i suoi reader"
  @ManyToMany(mappedBy = "readBooks", fetch = FetchType.LAZY)
  // LAZY: milioni di reader, mai caricare tutto insieme
  private List<Reader> readers;
}
@Entity
public class Reader implements Serializable {
  @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
  private int id;
  private String name;

  // Reader → Books letti: ManyToMany (Reader è owner: ha la JoinTable)
  @ManyToMany
  @JoinTable(name = "reader_book",
    joinColumns = {@JoinColumn(name = "reader_id")},
    inverseJoinColumns = {@JoinColumn(name = "book_id")})
  private List<Book> readBooks;  // owner
}

10. Esempio — Movie/Artist (TDE Luglio 2023)

Artist può dirigere film (1:N, Artist è il direttore) e recitare in film (N:M). Ha premi (N:M). L'app mostra i film diretti e recitati di un artista, in ordine cronologico decrescente.

@Entity
public class Artist implements Serializable {
  @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
  private int id;
  private String firstname, lastname, role;

  @Basic(fetch = FetchType.LAZY)
  @Lob
  private byte[] photo;  // LAZY: le foto sono grandi, caricare solo se serve

  // Film diretti da questo artista
  @OneToMany(mappedBy = "director", fetch = FetchType.EAGER)
  @OrderBy("issuedate DESC")
  // EAGER: pochi film per regista (curriculum breve), sempre mostrati
  private List<Movie> directedMovies;

  // Film in cui recita (come attore)
  @ManyToMany(mappedBy = "actors", fetch = FetchType.EAGER)
  @OrderBy("issuedate DESC")
  // EAGER: pochi film nel curriculum di un attore
  private List<Movie> actedMovies;

  // Premi vinti
  @ManyToMany(mappedBy = "artists", fetch = FetchType.EAGER)
  // EAGER: pochi premi per artista (ordini di grandezza: decine)
  private List<Award> awards;
}

@Entity
public class Movie implements Serializable {
  @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
  private int id;
  private String title, genre;
  private int revenues;
  @Temporal(TemporalType.DATE)
  private Date issuedate;

  // Il regista del film — ManyToOne, Movie è owner
  @ManyToOne
  @JoinColumn(name = "director_id")
  private Artist director;  // owner

  // Attori del film — ManyToMany, Movie è owner
  @ManyToMany
  @JoinTable(name = "actor_movie",
    joinColumns = {@JoinColumn(name = "movie_id")},
    inverseJoinColumns = {@JoinColumn(name = "actor_id")})
  private List<Artist> actors;  // owner
}
Motivazioni dei fetch type
  • directedMovies EAGER: un regista dirige pochi film (decine al massimo), e l'app li mostra sempre nella pagina dell'artista → vale la pena caricarli subito.
  • actedMovies EAGER: stessa logica — un attore recita in pochi film nel suo curriculum.
  • photo LAZY: le foto sono oggetti grandi. Caricarle ogni volta che si carica un Artist appesantirebbe tutte le query. Si caricano solo quando l'utente vede la pagina del singolo artista.
